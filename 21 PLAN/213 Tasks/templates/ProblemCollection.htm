Abstract

This document contains a collection of  problems and issues which should be addressed during the detailed item analysis. To make the amount of possible problems a little bit more handy, the Programme Office has defined Problem Types (PT) (268) to group the types of problems logically. Further each problem in the list has got a uniqe problem collection   number (PC#) by which the problem can be referenced when mentioned in your own problem and item analysis reports.

Be remainded that the problems' type and collection numbers are defined by the Programme Office and must not be modified. This aimes to provide global consistentency so that real problems which may occure in KUONI organisations can be addressed in the global Table of Occurence (311) or in  the inventory database (Web2000).

Please report new problems you would like to add to this list via cc-mail to the Y2K Programme Office and keep in mind that the Programme Office will update this list regularly.

Note: Any checklist is NOT, however, a specification. The checklist is used to ensure thoroughness and consistency in the assessment. It does not indicate completness nor does it constitutes compliance.

P001 C   Many orgnanisations have 'date difference' programs. These programs take various forms but generally fall into the
  • How many days (or working days) between date 1 and date 2
  • What is the date of date 1 plus n days

category. Just imagine what happens to them in 1999 or even earlier, if the number of days is more than 365.
At least those of you with such programs will get some advance warning of the perils of the year 2000.

 1.0
P002 A    

There are also programmers who like to use arrays for years, months or days; perhaps even combinations of them. Often these arrays have an arbitrary 'start' date.

An example is that each year is an element in an array.

We NEVER have data before '89 (it's catching this 2 digit year stuff) so we take the year number, subtract 88 and then we have our element number.
Another alternative is that we have our data in date order and we take the year contained in the first record minus one and then we use that as the base of our array.
Now you must have realised what will happen.

  • 96-88 gives element number 7,
  • 99-88 gives element number 10,
  • 00-88 gives element number -88 (minus eighty eight).

    If you have a program that builds an array of data where the year is used as an index for the array elements, don't forget that it does not have to be in the year number itself, it could be the year and the Julian day number, or the week number used to determine the array element number.
    You are very likely to find out sometime during 1999 at the latest that you start addressing a non-existant element number except some compilers do not check the number dynamically, they just convert the element to an address and merrily overwrite some other part of storage or attempt to use unallocated storage.
    Any sensible system wil not allow the second to happen, but no doubt some of the PC operating systems are less efficient at this.

PO13 99   One place where all 9's does show up is in the use of a date complement (not
compliment, thank you). In this case it is the "inverse" of the date. Date
complements are used to reverse sort data so that the most current records
are shown at the top on screens and reports. It is commonly used and
sometimes all 9's could be compared to show records that should actually be
at the bottom of the list. In the case where no date, a "null" date, or a
date in the format of 00/00/00 was entered, the possible calculation to
produce a "9's complement" would be something like:

In non-Y2K Ready format:
DATE COMPLEMENT
999999 - 980219 = 019780
999999 - 000102 = 999897
999999 - 000000 = 999999

This would force the second date to be sorted AFTER the first (because its
value is lower) which would mess up the reason that you were doing a
"reverse" sort of putting the "newer" stuff ahead of the old.

By getting explicit with the dates, you avoid the problem:
DATE COMPLEMENT
99999999 - 20000102 = 79999897
(this next one might not be a real date but your "windowing" logic will try
to convert it to one!!!)
99999999 - 20000000 = 79999999
99999999 - 19980219 = 80019780

So... 9's are out there and you have to watch for this. If you aren't
checking for this type of processing, you'd better go back and have another
look. The variable names to store the complement may not look like date
fields and many find/fix tools may not comprehend what is being done.
